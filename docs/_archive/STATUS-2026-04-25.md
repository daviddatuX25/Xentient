## 1. System Overview

Xentient is a 3-tier IoT system: **ESP32 hardware** (firmware in C++) → **Harness** (Node.js/TypeScript core + brain) → **Dashboard** (HTTP/SSE browser UI).

```
┌──────────────┐    MQTT/WS     ┌────────────────────┐    MCP/stdio    ┌──────────────┐
│  ESP32-CAM   │◄─────────────►│  Core (Node.js)     │◄──────────────►│  brain-basic  │
│  + Node Base │    192.168.x   │  MCP Server         │   child proc   │  (STT→LLM→TTS)│
│  + sensors   │               │  RuleEngine (TODO)  │               └──────────────┘
└──────────────┘               │  HealthMonitor(TODO)│
                               │  AudioServer (WS)    │    HTTP/SSE    ┌──────────────┐
                               │  CameraServer (WS)   │◄─────────────►│  Dashboard    │
                               │  ModeManager         │               │  (browser)    │
                               │  ControlServer       │               └──────────────┘
                               │  MqttClient          │
                               └──────────────────────┘
```

### 2.2 What's Broken in Production Firmware

| # | Issue | Priority | Impact |
|---|-------|----------|--------|
| **FW-1** | PIR ISR not in main.cpp | P0 | `attachInterrupt` for GPIO13 exists in validation sketches but NOT in `main.cpp`. The PIR ISR code is missing. Sleep→listen is non-functional in production. |
| **FW-2** | MQTT callback doesn't dispatch | P1 | `mqtt_callback` logs hex but never parses mode_set or display_update messages. LCD never changes from commands. |
| **FW-3** | No I2S speaker output | P1 | MAX98357A works in validation but main firmware has no I2S TX driver. TTS audio from harness cannot be played. I2S_NUM_0 conflict (shared BCLK/LRCK for both mic and speaker). |
| **FW-4** | No TTS audio consumption | P1 | WebSocket client only sends outbound audio. Never processes incoming binary. `xentient/audio/out` is defined but unimplemented. |
| **FW-5** | Pipeline status never published | P2 | `xentient/pipeline/state`, `xentient/session/complete`, `xentient/session/error` are defined in messages.h but never published by firmware. |
| **FW-6** | Camera MQTT never used | P2 | `xentient/camera/request` and `xentient/camera/status` defined but firmware never publishes or subscribes. Camera relay is purely UART push. |
| **FW-7** | Hardcoded network credentials | P2 | WiFi SSID ("your-ssid"), password, MQTT broker IP, WS harness IP all hardcoded in `messages.h`. No provisioning. |
| **FW-8** | I2S full-duplex conflict | P1 | BCLK=26 and LRCK=25 shared between INMP441 (RX) and MAX98357A (TX). Both on I2S_NUM_0. Only RX driver is installed. Adding TX on same port causes conflict. Solutions: (a) I2S_NUM_1 for speaker, (b) time-division multiplexing, (c) separate I2S ports. |

### 2.3 Firmware Validation Sketches

11 PlatformIO environments (1 production + 10 validation). Only `node_base` and `val_pir_hwtest` have compiled binaries. Others need `pio run -e <env>` before flashing.

## 8. Data Flow (End-to-End, What Actually Works)

### Voice Pipeline (WORKS)

```
User speaks
  → INMP441 mic (I2S RX, 16kHz mono)
  → ESP32 VAD (RMS hysteresis: enter 1000, exit 600)
  → VAD start: publish trigger_pipeline{source:voice,stage:start} on MQTT
  → Harness: mqtt.on("triggerPipeline") → MCP notification voice_start
  → Harness: AudioAccumulator buffers PCM chunks
  → VAD end: publish trigger_pipeline{source:voice,stage:end} on MQTT
  → Harness: mqtt.on("triggerPipeline") → flush buffer → MCP notification voice_end
  → brain-basic: receives voice_end → STT → LLM → TTS
  → brain-basic: calls xentient_play_audio(MCP)
  → Harness: AudioServer.sendAudio() → WS binary to ESP32
  → [BROKEN] ESP32: no I2S TX driver → audio never plays
```

### Motion Wake (BROKEN IN PRODUCTION)

```
PIR triggers on GPIO13
  → [BROKEN] No attachInterrupt in main.cpp
  → If manually triggered (web button): publish mode_set{mode:listen}
  → Harness: ModeManager.transition("listen")
  → Harness: MCP notification mode_changed
  → brain-basic: logs mode change
```

### Camera (WORKS)

```
ESP32-CAM captures QQVGA frame every 3s
  → UART2 chunked protocol (CRC-8) → Node Base
  → Node Base reassembles → WS binary (0xCA prefix) → Harness
  → Harness: AudioServer discriminates → CameraServer → dashboard clients
  → brain-basic: can call xentient_capture_frame() → gets JPEG base64
```

### Sensor Telemetry (WORKS)

```
BME280 reads every 5s
  → MQTT xentient/sensors/env
  → Harness: MqttClient → ModeManager.handleSensorEvent() → sensorCache
  → Harness: MCP notification sensor_update
  → brain-basic: logs sensor data
  → Dashboard: GET /api/sensors → returns sensorCache
```

## 9. Critical Path to Demo

The minimum path to a working demo, in order:

### Blocker 1: PIR Wake (P0)
- **Fix:** Add `attachInterrupt(digitalPinToInterrupt(PIN_PIR_INT), pir_isr, RISING)` to `main.cpp` `setup()`
- **Add:** ISR handler that publishes `sensor_data{peripheralType:0x11, payload:{motion:true}}` to `xentient/sensors/motion`
- **Test:** Stand in front of sensor → mode transitions from sleep to listen → LCD changes

### Blocker 2: Audio Output (P1)
- **Fix:** Resolve I2S full-duplex conflict (Options: I2S_NUM_1 for speaker, or time-division multiplexing)
- **Add:** I2S TX driver in main firmware that consumes `xentient/audio/out` WebSocket binary
- **Add:** 0xA0 prefix handling on ESP32 receive side
- **Test:** Speak → STT → LLM → TTS → hear response from speaker

### Blocker 3: MQTT Inbound Dispatch (P1)
- **Fix:** Parse `xentient/control/mode` and `xentient/display` in `mqtt_callback`
- **Add:** Mode transition on mode_set messages, LCD update on display_update messages

### Blocker 4: Harness Audio Prefix (P1)
- **Fix:** Prepend `Buffer.from([0xA0])` in `AudioServer.sendAudio()`
- **Already identified, simple one-line fix**

- Power path assembly (18650 → TP4056 → MT3608 → 5V/3.3V)

## 11. What the Senior Developer Needs to Decide

From the spec's 7 decision points, these are the ones that affect code NOW:

| Decision | Impact if wrong | Recommendation |
|----------|-----------------|---------------|
| **Rule trigger schema** (D-schedule) | If wrong, both RuleEngine and Hermes SkillRegistry need rewriting | Nail down `RuleTrigger` and `RuleAction` types before coding RuleEngine |
| **Rule tick interval** (D-interval) | Too fast wastes CPU, too slow misses events | Default 2s, configurable per-rule via `minIntervalMs` |
| **Rule persistence** (D-persist) | Lost rules on Core restart | In-memory only for demo. Document as known limitation. |
| **Failover mode** (D-failover) | Demo crashes if Hermes disconnects | Default to rule-only mode. No brain-basic spawn in demo. |
| **I2S full-duplex resolution** (FW-8) | Speaker won't work in production | Try I2S_NUM_1 for MAX98357A first. If conflicts, fall back to time-division. |

The other 2 decisions (SSE+stdio dual transport, brain-basic as standalone) don't affect Phase 2 code. They matter for Phase 3-4.
