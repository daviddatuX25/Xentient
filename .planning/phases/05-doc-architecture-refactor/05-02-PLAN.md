**docs/CONTRACTS.md** — Authoritative wire format source (per D24):

1. Header: "# Xentient Wire Contracts\n\n> L2 Spec — Authoritative source for all message formats, MQTT topics, and data contracts between Node Base, Harness, and integrations. If contracts.ts and this doc disagree, this doc wins until contracts.ts is updated."

2. Section "Versioning" — Every packet has `v: 1` at root level. Harness rejects mismatched versions with `{error:"version_mismatch"}`.

3. Section "Payload Cap" — 3KB hard limit on any MQTT packet. Binary audio over WebSocket has no cap.

4. Section "Key Naming Convention" — camelCase everywhere. ArduinoJson handles both sides.

5. Section "Timestamps" — `epoch-millis uint32` everywhere (half the bytes of ISO8601, demo-safe).

6. Section "Enum Widths" — All opcodes pinned to `uint8_t` (0-255). TS mirrors with `z.number().int().min(0).max(255)`.

7. Section "Peripheral ID Registry" — Table: 0x10=Speaker, 0x11=PIR, 0x12=BME280, 0x13=INMP441, 0x14=ESP32-CAM, 0x15=LCD.

8. Section "MQTT Topic Map" — Table of all topics:
   - `xentient/audio/in` — audio chunks from mic
   - `xentient/audio/out` — TTS audio to speaker
   - `xentient/sensors/env` — environmental sensor data
   - `xentient/sensors/motion` — PIR motion events
   - `xentient/display` — LCD display commands
   - `xentient/display/faces` — LCD face table for pack
   - `xentient/control/pack` — pack switch/reload commands
   - `xentient/status/packs` — pack status responses
   - `xentient/control/space` — space switch commands
   - `xentient/status/space` — space status responses
   - `xentient/control/mode` — mode set commands
   - `xentient/status/mode` — mode status responses
   - `xentient/pipeline/state` — pipeline state transitions
   - `xentient/session/error` — session errors (recoverable vs fatal)

9. Section "Message Schemas" — JSON examples for each message type:
   - Audio chunk: {v:1, type:"audio_chunk", sessionId, sequence, payload:base64}
   - Display update: {v:1, type:"display_update", mode:"expression"|"text"|"status", line1, line2, duration?:ms}
   - Space status: {v:1, type:"space_status", spaces:[...]}
   - Mode command: {v:1, type:"mode_set", mode:"sleep"|"listen"|"active"|"record"}
   - Pack switch: {v:1, type:"pack_switch", name:"pack-name"}
   - Session error: {v:1, type:"session_error", recoverable:bool, message:string}
   - Sensor data: {v:1, type:"sensor_data", peripheralType:enum, payload:{...}}

10. Section "Zod Schema Convention" — All schemas defined in `harness/src/shared/contracts.ts` using Zod. This doc is the design reference; contracts.ts is the runtime enforcement.

**docs/PACKS.md** — Pack folder spec (per D25, NOTES.md Platform Vision section):

1. Header: "# Xentient Pack System\n\n> L2 Spec — Pack folder specification, manifest schema, handler enum, lifecycle, and guardrails. A pack is a folder that defines a bot's behavior."

2. Section "Concept" — "Behavior is a folder." One pack = one persona + tools + voice + memory rules. Swap the folder, swap the brain.

3. Section "Folder Structure":
   ```
   packs/
     <pack-name>/
       manifest.json       # required — Zod-validated
       persona.md          # required — system prompt + optional LCD face overrides
       tools.json          # optional — tool/function defs + MQTT topic bindings
       memory-rules.json   # optional — fact-extraction + injection policy
       voice.json          # optional — TTS provider + voice ID + speed/pitch
       assets/             # optional — custom LCD bitmap chars, audio cues
   ```

4. Section "Manifest Schema" — Full JSON schema for manifest.json:
   - v, name (kebab-case), displayName, author, version, description, capabilities array, compatibility object (harness version, hardware array), entrypoint object (persona, tools, memory, voice)
   - Copy from NOTES.md manifest.json example.

5. Section "Persona Format" — persona.md structure: YAML front-matter (voice, style, greeting, lcdFaces override table), body = system prompt with {{memory.userName}} and {{sensor.temp}} interpolation.

6. Section "Handler Enum" — Table of all handler types with "What It Does" and "Available When" columns:
   - mqtt-publish, mqtt-request-response, basic-llm, hermes-chat, hermes-memory, hermes-skill, computer-use, agent-delegate
   - Copy from repurpose.md Tool Router table (per D25, D5).

7. Section "tools.json Structure" — Array of tool defs with OpenAI-compatible schema. Each tool has a `handler` field that must be one of the enum values above. Example with mqtt-publish handler.

8. Section "memory-rules.json Structure" — extract, inject, retention fields. Copy from NOTES.md.

9. Section "voice.json Structure" — provider, voiceId, speed, pitch, fallback object. Copy from NOTES.md.

10. Section "Lifecycle" — Boot → Load → Validate → Register. Hot-reload via chokidar with 500ms debounce. MemoryDB preserved across reloads. Pack switch via MQTT. Validate fail falls back to `default` pack.

11. Section "Guardrails" (per D28 and NOTES.md junior-dev guardrails):
   - Pack names: `[a-z0-9-]{1,32}` only, kebab-case

   - Asset files >1MB rejected
   - `default` pack is reserved and always present
   - One pack active at a time — no stacking, no inheritance, no composition
   - Adding a handler type requires a harness PR — intentional friction
