# 08-07: ControlServer Refactoring

## Goal

Replace the hand-rolled `if/else` routing in `ControlServer.handleRequest()` with a zero-dependency `MicroRouter` class that supports method matching, path parameters (`:id`), and structured error responses. Expand the constructor dependency injection to include `SpaceManager`, `EventBridge`, `PackLoader`, `SensorHistory`, and `SkillLog` (same pattern as `McpToolDeps`).

## Why First (Wave 1 Blocker)

All new REST endpoints (08-01) and SSE events (08-02) depend on:
1. A route table that can handle ~20 routes cleanly
2. ControlServer having access to SpaceManager, PackLoader, EventBridge, SkillLog, SensorHistory

Without this refactoring, adding 15 new endpoints to the existing `if/else` chain becomes unmaintainable.

## Resolved Prerequisites (from Phase 7 post-implementation)

The following were previously listed as blockers but are **already resolved**:
- **SkillLog class** — Already exists at `harness/src/engine/SkillLog.ts`, imported by SkillExecutor and SpaceManager. SpaceManager exposes `this.skillLog = new SkillLog()`.
- **SpaceManager.listSkills()** — Already exists at `SpaceManager.ts:124`, iterates all executors and returns flat skill array.

## Implementation

### Step 1: Create `MicroRouter` class

**File:** `harness/src/comms/MicroRouter.ts` (new, ~70 lines)

```typescript
type RouteHandler = (req: IncomingMessage, res: ServerResponse, params: Record<string, string>) => Promise<void>;

interface Route {
  method: string;
  pattern: RegExp;
  paramNames: string[];
  handler: RouteHandler;
}

export class MicroRouter {
  private routes: Route[] = [];

  add(method: string, path: string, handler: RouteHandler): this {
    const paramNames: string[] = [];
    const patternStr = path.replace(/:(\w+)/g, (_m, name) => {
      paramNames.push(name);
      return `(?<${name}>[^/]+)`;
    });
    const pattern = new RegExp(`^${patternStr}/?$`);
    this.routes.push({ method: method.toUpperCase(), pattern, paramNames, handler });
    return this;
  }

  resolve(method: string, url: string): { handler: RouteHandler; params: Record<string, string> } | { status: number } {
    const pathname = url.split('?')[0];
    const methodUpper = method.toUpperCase();
    let pathMatched = false;

    for (const route of this.routes) {
      const match = route.pattern.exec(pathname);
      if (!match) continue;
      pathMatched = true;
      if (route.method !== methodUpper) continue;

      const params: Record<string, string> = {};
      if (match.groups) {
        for (const name of route.paramNames) {
          if (match.groups[name] !== undefined) params[name] = match.groups[name];
        }
      }
      return { handler: route.handler, params };
    }
    return { status: pathMatched ? 405 : 404 };
  }
}
```

Key design decisions:
- Named capture groups (`(?<id>[^/]+)`) for parameter extraction
- 404 vs 405 distinction (path matched but method didn't → 405 Method Not Allowed)
- Fluent API (`router.add().add().add()`)
- Zero external dependencies

**Critical: `this` binding on handlers (Expansion 7.1)**

When passing `this.handleListSkills` to `router.add()`, the `this` context is lost at the call site. Every handler registration **MUST** use `.bind(this)`:

```typescript
// WRONG — silent runtime bug, this.deps is undefined when handler fires:
this.router.add('GET', '/api/skills', this.handleListSkills)

// CORRECT:
this.router.add('GET', '/api/skills', this.handleListSkills.bind(this))
```

Alternative: declare handlers as arrow function properties (auto-binds `this`):
```typescript
private handleListSkills = async (req: IncomingMessage, res: ServerResponse, params: Record<string, string>): Promise<void> => {
  const skills = this.deps.spaceManager.listSkills();
  this.sendJSON(res, 200, skills);
};
```

**Decision:** Use `.bind(this)` for consistency with existing method style. Add a lint comment:
```typescript
// IMPORTANT: All handlers MUST be .bind(this)'d — MicroRouter calls them detached from `this`
```

### Step 4: Request resolution order — static file fallthrough (Expansion 7.2)

The request resolution order must be:

1. **CORS preflight** → `OPTIONS` returns 204 immediately
2. **SSE** → `/api/events` bypasses router (long-lived connection)
3. **API routes** → MicroRouter resolves `/api/*` paths
4. **Static files** → If router returns 404 for a **non-`/api/`** path, try static file serving
5. **404** → No match anywhere

```typescript
private async handleRequest(req: IncomingMessage, res: ServerResponse): Promise<void> {
  const method = req.method ?? 'GET';
  const url = req.url ?? '/';

  // 1. CORS
  this.setCORSHeaders(res);
  if (method === 'OPTIONS') { res.writeHead(204); res.end(); return; }

  // 2. SSE (long-lived, skip router)
  if (url.split('?')[0] === '/api/events') { this.handleSSE(req, res); return; }

  // 3. API routes
  const result = this.router.resolve(method, url);
  if ('handler' in result) {
    try {
      await result.handler(req, res, result.params);
    } catch (err) {
      this.sendJSON(res, 500, { error: 'Internal server error' });
    }
    return;
  }

  // 4. Static files — only for non-API paths
  const pathname = url.split('?')[0];
  if (!pathname.startsWith('/api/')) {
    this.serveStatic(req, res);
    return;
  }

  // 5. 404 for API paths that didn't match
  const msgs: Record<number, string> = { 404: 'Not found', 405: 'Method not allowed' };
  this.sendJSON(res, result.status, { error: msgs[result.status] ?? 'Bad request' });
}
```

### Step 5: Request body size limit (Expansion 7.4)

No guard against giant POST bodies currently. Add a max body size to prevent memory exhaustion:

```typescript
private static MAX_BODY_SIZE = 64 * 1024; // 64KB

private async parseBody(req: IncomingMessage): Promise<unknown> {
  return new Promise((resolve, reject) => {
    const chunks: Buffer[] = [];
    let totalSize = 0;
    req.on('data', (c) => {
      totalSize += c.length;
      if (totalSize > ControlServer.MAX_BODY_SIZE) {
        reject(new Error('Body too large'));
        req.destroy(); // Abort the connection to stop receiving data
        return;
      }
      chunks.push(c);
    });
    req.on('end', () => {
      const body = Buffer.concat(chunks).toString();
      if (!body) { resolve({}); return; }
      try { resolve(JSON.parse(body)); }
      catch (e) { reject(new Error('Invalid JSON')); }
    });
    req.on('error', reject);
  });
}
```

Handle in the request pipeline:
```typescript
try {
  await result.handler(req, res, result.params);
} catch (err) {
  if ((err as Error).message === 'Body too large') {
    this.sendJSON(res, 413, { error: 'Request body too large (max 64KB)' });
  } else {
    this.sendJSON(res, 500, { error: 'Internal server error' });
  }
}
```

## Edge Cases

- **Trailing slashes:** `/api/skills/` and `/api/skills` must both resolve to the same route. The regex pattern `^/api/skills/?$` handles this.
- **Query strings:** `url.split('?')[0]` strips query params before routing.
- **CORS preflight:** `OPTIONS` requests must bypass the router and return 204 immediately.
- **SSE endpoint:** Must bypass the router entirely (long-lived connection).
- **Static file fallthrough:** Non-`/api/` paths that don't match a route fall through to static file serving. Only `/api/*` paths that don't match get a JSON 404.
- **Body too large:** If client sends >64KB, connection is destroyed mid-stream. Client sees connection reset, not a clean 413. This is acceptable — legitimate requests won't exceed 64KB.
- **Handler `this` binding:** Every `.add()` call must use `.bind(this)`. Missing bind causes `this.deps` to be `undefined` at runtime — a silent bug that only manifests when the route is actually hit.
