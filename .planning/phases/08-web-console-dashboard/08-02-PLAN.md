# 08-02: SSE Event Expansion

## Goal

Expand the SSE endpoint (`/api/events`) to push all events the dashboard needs for real-time updates. Add lifecycle events for skills, packs, and event mappings. Add throttled sensor updates and counter updates.

## Prerequisites

- **08-07 complete** — ControlServer must accept SpaceManager, EventBridge, PackLoader, SkillLog as deps

## New SSE Event Types

| Event Type | Trigger | Payload |
|------------|---------|---------|
| `skill_registered` | `SpaceManager.registerSkill()` | `{ skillId, source, triggerType }` |
| `skill_removed` | `SpaceManager.removeSkill()` | `{ skillId }` |
| `skill_updated` | `SpaceManager.updateSkill()` / `disableSkill()` | `{ skillId, patch }` |
| `pack_loaded` | `PackLoader.loadPack()` | `{ packName, skillCount }` |
| `pack_unloaded` | `PackLoader.unloadCurrentPack()` | `{ packName }` |
| `event_mapping_added` | `EventBridge.addCustomMapping()` | `{ mappingId, source, eventName }` |
| `event_mapping_removed` | `EventBridge.removeMapping()` | `{ mappingId }` |
| `sensor_update` | sensorCache change | `{ temperature, humidity, pressure }` (throttled 1/s) |
| `counter_update` | SkillExecutor counter change | `{ counters: Record<string, number> }` (throttled 1/s) |

## Implementation

### Step 1: Add EventEmitter events to SpaceManager

**File:** `harness/src/engine/SpaceManager.ts`

Currently, SpaceManager emits `skill_fired`, `skill_escalated`, `skill_conflict` via `broadcastObservabilityEvent()`. Add new lifecycle events:

```typescript
// In registerSkill(), after skill is added:
this.emit('skill_registered', { skillId: skill.id, source: skill.source, triggerType: skill.trigger.type });

// In removeSkill(), before skill is removed:
this.emit('skill_removed', { skillId });

// In updateSkill(), after patch is applied:
this.emit('skill_updated', { skillId, patch });

// In disableSkill(), after enabled state is toggled:
this.emit('skill_updated', { skillId, patch: { enabled } });
```

import { EventEmitter } from 'events';

export class PackLoader extends EventEmitter {

### Step 5: Add throttled SSE broadcaster

**File:** `harness/src/comms/ControlServer.ts`

Add a `ThrottledSSEBroadcaster` class (or inline throttling in ControlServer):

```typescript
private sensorThrottleTimer: NodeJS.Timeout | null = null;
private lastSensorBroadcast = 0;
private counterThrottleTimer: NodeJS.Timeout | null = null;
private lastCounterBroadcast = 0;
private static THROTTLE_MS = 1000; // 1 second

private broadcastThrottledSensor(data: object): void {
  const now = Date.now();
  if (now - this.lastSensorBroadcast >= ControlServer.THROTTLE_MS) {
    this.lastSensorBroadcast = now;
    this.broadcastSSE({ type: 'sensor_update', ...data });
  } else if (!this.sensorThrottleTimer) {
    this.sensorThrottleTimer = setTimeout(() => {
      this.sensorThrottleTimer = null;
      this.lastSensorBroadcast = Date.now();
      this.broadcastSSE({ type: 'sensor_update', ...data });
    }, ControlServer.THROTTLE_MS - (now - this.lastSensorBroadcast));
  }
}
```

## Verification

1. `npx tsc --noEmit` — Zero type errors
2. `npx vitest run` — All tests pass
3. Register a skill via REST: `curl -X POST /api/skills ...` → SSE client receives `skill_registered` event within 1 second
4. Remove a skill: `curl -X DELETE /api/skills/xyz` → SSE client receives `skill_removed` event
5. Load a pack: `curl -X POST /api/packs/default/load` → SSE client receives `pack_loaded` event
6. Sensor update arrives → SSE client receives throttled `sensor_update` (max 1/second)
7. Counter update → SSE client receives `counter_update` (max 1/second)

## Edge Cases

- **SSE client connects mid-stream:** Client receives `connected` event, then must fetch REST state for initial render. New lifecycle events only fire going forward.
- **Multiple SSE clients:** `broadcastSSE` iterates all clients; throttling is per-server (not per-client), so all clients get the same throttled rate.
- **PackLoader events with no listeners:** EventEmitter default is 10 listeners; if more are added, increase with `packLoader.setMaxListeners(20)`.
- **Counter update interval:** 1s polling is simple. If counters change faster, the dashboard still gets updates at 1s granularity — acceptable for v1.
